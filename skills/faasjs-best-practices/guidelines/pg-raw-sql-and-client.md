# PG Raw SQL and Client Guide

When `@faasjs/pg` needs SQL outside the fluent builder, default to `client.raw(...)` with parameters, and only use raw fragments deliberately.

## Use This Guide When

- writing `client.raw(...)` queries
- choosing between `getClient()` and `createClient(...)`
- building custom joins, predicates, or schema statements that the builder cannot express directly
- reviewing query logging or transaction boundaries

## Default Workflow

1. Set `DATABASE_URL` in the environment and prefer `getClient()` so app code and tests share one bootstrap path.
2. Prefer `client.query(...)` when the fluent API already supports the query.
3. Use `client.raw(...)` with tagged template strings or `?` placeholders for custom SQL.
4. Use `escapeIdentifier(...)` or `rawSql(...)` only for trusted identifiers or SQL fragments that cannot be parameterized.
5. Wrap multi-step data mutations in `client.transaction(...)`.

## Minimal Example

```ts
import { getClient } from '@faasjs/pg'

const client = getClient()

await client.transaction(async (trx) => {
  await trx.raw('UPDATE users SET name = ? WHERE id = ?', 'Alice', 1)
  await trx.raw`INSERT INTO audit_logs (action, user_id) VALUES (${'rename_user'}, ${1})`
})
```

## Rules

### 1. Parameterize values by default

- Use placeholders or template parameters for runtime values.
- Prefer builder helpers or `client.raw(...)` over manual string concatenation.
- Treat value interpolation in SQL strings as a bug unless the input is a trusted static literal.

### 2. Keep database bootstrap consistent across environments

- Prefer `getClient()` for the default application client so the shared bootstrap path stays tied to `process.env.DATABASE_URL`.
- Reach for `createClient(process.env.DATABASE_URL, options)` only when custom `postgres.js` options or multiple database connections are required.
- Treat `getClient()` throwing as a signal that the shared bootstrap path was not configured.
- In tests, let `TypedPgVitestPlugin()` populate `DATABASE_URL` instead of building a separate testing-only connection path.

### 3. Escape identifiers and fragments explicitly

- SQL identifiers cannot be parameterized; use `escapeIdentifier(...)` or a carefully bounded `rawSql(...)` fragment instead.
- `rawSql(...)` is for trusted SQL only and must not wrap untrusted user input.
- Keep the trusted boundary small and obvious in code review.

### 4. Choose the right transactional boundary

- Use `client.transaction(...)` for multi-step DML or mixed read-write flows.
- Use `SchemaBuilder.run()` for batched DDL generated by schema helpers.
- Keep all-or-nothing behavior explicit in both review and tests.

### 5. Use query logging selectively

- The client logger is optional.
- Debug logging is appropriate for query timing, troubleshooting, or temporary diagnostics.
- Avoid coupling normal application logic to debug logging side effects.

### 6. Move repeated raw SQL back toward shared helpers or the builder

- If a raw query becomes common or reusable, consider whether it belongs in a shared helper or in the fluent query surface instead.
- Prefer one reviewed abstraction over many near-duplicate raw SQL snippets.
- When `@faasjs/pg` grows a clause helper that covers the same case, update app code to leave raw SQL behind.

## Review Checklist

- values are parameterized
- the default client bootstrap goes through `getClient()` and `process.env.DATABASE_URL`
- identifiers or SQL fragments are escaped or explicitly trusted
- raw SQL is used only where the builder surface is insufficient
- multi-step writes use `transaction(...)` when atomicity matters
- debug logging changes do not affect normal runtime behavior
