# faas.yaml Configuration Specification

## Background

`faas.yaml` is the runtime configuration entry used by FaasJS for config loading, local dev server parsing, and type generation.

This specification defines the public baseline that aligns with current FaasJS runtime behavior.

Related references:

- `packages/node-utils/src/load_config.ts`
- `packages/node-utils/src/parse_yaml/index.ts`
- `packages/dev/src/server_config.ts`
- `packages/dev/src/typegen.ts`
- `parseYaml()` in `@faasjs/node-utils`

## Goals

- Keep config discovery order and merge order predictable.
- Keep staging-level validation behavior predictable.
- Explicitly define the supported YAML subset.
- Provide a parser entry point for custom Node.js tools that support the same YAML subset.

## Normative Rules

### 1. File Naming and Placement

1. The config file MUST be named `faas.yaml`.
2. Project-level config SHOULD be placed at `<projectRoot>/src/faas.yaml`.
3. Additional `faas.yaml` files MAY be placed in nested directories under `src/` for local overrides.

### 2. Discovery and Merge Order

1. For an API file at `<srcRoot>/<path>/<file>.api.ts`, the loader MUST discover config files from shallow to deep:
   - `<srcRoot>/faas.yaml`
   - `<srcRoot>/<path-segment-1>/faas.yaml`
   - ...
   - `<srcRoot>/<path>/faas.yaml`
2. Valid configs MUST be merged in discovery order, with deeper configs overriding shallower ones.
3. For each staging key where `S != defaults`, the effective staging config MUST be `deepMerge(defaults, S)` after file-level merge is complete.
4. If the requested staging does not exist, the runtime MUST fall back to `defaults`, then to an empty object.

### 3. Root Object and Staging Keys

1. The resolved root value MUST be an object when provided.
2. Each top-level staging value MUST be an object when provided.
3. `defaults` SHOULD exist and SHOULD contain shared baseline settings.

### 4. `server` Node Conventions

1. `<staging>.server` MUST be an object when present.
2. `<staging>.server.root` MUST be a string when present.
3. `<staging>.server.base` MUST be a string when present.
4. In `@faasjs/dev`, `server.root` MUST be resolved relative to the project root and represents the project root.
5. In `@faasjs/dev`, the source root MUST be `<server.root>/src`.
6. In type generation, the output path MUST be `<server.root>/src/.faasjs/types.d.ts`.

### 5. `plugins` Node Authoring

1. `<staging>.plugins` MAY be omitted.
2. The recommended form is a mapping from plugin key to plugin config object.
3. Plugin keys within the same staging SHOULD be stable, because `loadConfig()` derives the runtime `name` from the key.
4. `name` is a field generated by the loader at runtime and MUST NOT be hand-written in `faas.yaml`.
5. A plugin config object MAY contain `type`, `config`, and other plugin-specific fields.
6. Plugin `type` MAY be written as a built-in plugin type, a bare package type, a scoped package name, a relative path, an absolute path, or a `file://` local file URL.
7. Plugin-internal fields are outside the scope of this specification and MUST be preserved as-is during merge.

### 6. Supported YAML Subset

1. The parser MUST support indentation-based mappings and sequences.
2. The parser MUST support plain, single-quoted, and double-quoted scalars.
3. The parser MUST support these scalar values: `null`, `boolean`, `number`, and `string`.
4. The parser MUST support anchors (`&`), aliases (`*`), and the merge key (`<<`) in mappings.
5. The parser MUST NOT allow tabs as indentation.
6. The parser MUST NOT allow multi-document YAML (`---`, `...`).
7. The parser MUST NOT allow block scalars (`|`, `>`), YAML tags (`!`), or non-empty flow collections (`[a]`, `{a: 1}`).
8. Empty flow collections `[]` and `{}` MAY be used.

### 7. Error Handling

1. Invalid config structure MUST throw an error prefixed with:
   - `[loadConfig] Invalid faas.yaml <filePath> at "<keyPath>": <reason>`
2. Unsupported YAML syntax/features MUST throw a parse error prefixed with:
   - `[parseYaml]`
3. Config consumers (e.g., dev server startup or typegen) SHOULD fail fast on config loading failure.

## Example

```yaml
defaults:
  server:
    root: .
    base: /api
  plugins:
    http:
      type: http
      config:
        cookie:
          secure: false
          session:
            secret: replace-me
```
