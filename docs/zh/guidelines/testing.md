# 测试指南

在 FaasJS 项目中编写或审查测试时使用本指南。

## 适用场景

- 编写或审查 API、工具函数、hooks 或组件的测试
- 决定是否 mock 某个依赖
- 选择合适的测试级别（单元、集成或类型）
- 检查测试是否覆盖行为而非实现细节

## 默认工作流程

1. 从测试应捕获的公共行为或回归问题开始。
2. 选择能够仍然执行真实边界的最小测试级别。
3. 保持本地业务逻辑、辅助函数、hooks 和组件为真实实现，除非明确的边界需要隔离。
4. 仅 mock 外部、非确定性、昂贵或难以控制的边界。
5. 保持 mock 设置明确、限定在场景范围内，且比其替代的真实行为更简单。
6. 覆盖成功路径以及调用方实际依赖的失败或状态转换路径。
7. 在每个用例之间重置共享全局状态、定时器、环境变量和 mocks。
8. 将每个测试文件放置在它所保护的功能、API、hook、组件、辅助函数或任务附近。
9. 在交接前，运行 `vp check --fix` 和 `vp test` 作为验收关口；如果当前环境中无法运行任一命令，请记录阻塞因素以及仍然运行了的更窄范围的验证。

## 规则

### 1. 测试公共行为而非实现细节

- 断言返回的数据、渲染输出、抛出的错误、写入的响应字段或其他外部可见的效果。
- 避免断言内部调用顺序、私有辅助函数的使用、中间状态形状或框架内部实现，除非这就是约定的行为。
- 保持行为不变的重构通常不应需要重写测试。

### 2. 优先使用被测边界内的真实依赖

- 当可以通过公共输入到达场景时，保持本地函数、hooks、子组件、模式和辅助函数为真实实现。
- 这样可以更早地暴露依赖和集成回归问题，而不是将它们隐藏在测试替身之后。
- 不要仅仅为了让测试更短而进行 mock，如果这样做会移除生产中最可能出问题的部分。

### 3. 仅 mock 必须隔离的最窄边界

- 好的 mock 边界通常是网络调用、数据库访问、时钟、随机值、文件系统、进程环境变量或第三方服务。
- 如果低级别测试需要大量内部 mock 才能工作，请升级到更高级别的测试。
- 当需要 mock 时，使约定明确，并保持伪造的行为比生产行为更简单。

### 4. 使用能建立信心的最小测试级别

- 对隔离的逻辑使用纯函数或单元测试。
- 对接口约定、验证和 HTTP 行为使用 handler 或 API 测试。
- 对 UI 行为使用组件或 hook 测试。
- 优先选择能练习真实集成而不需要大量内部 mock 的测试级别。

### 5. 覆盖用户和调用方依赖的路径

- 成功路径
- 有意义的验证或错误路径
- 集成在生产中可能失败的边界故障
- 功能支持时的重载、重试或状态转换行为
- 涉及共享状态时的清理或重置行为

### 6. 将测试放在它们保护的项目区域附近

- 不要将所有包测试集中到 `src/__tests__` 目录中，包括该集中目录下以功能命名的子文件夹。
- 按项目用途、功能或切片拆分代码和测试，使 API、UI、任务、辅助函数和集成测试归属于拥有该行为的 feature 文件夹。
- 将测试放在所属文件夹的 `__tests__` 中，例如 `list.api.ts` 的测试放在 `src/features/users/api/__tests__/list.test.ts`，或 `cleanup.job.ts` 的测试放在 `src/features/users/jobs/__tests__/cleanup.test.ts`。
- 当被保护的业务代码本身是单个文件时，将其转换为包含 `index.ts` 或 `index.tsx` 的文件夹，并将测试放在该文件夹的 `__tests__` 下，例如 `src/useBilling/index.ts` 和 `src/useBilling/__tests__/useBilling.test.ts`。

## 审查清单

- 测试断言公共行为
- 本地依赖保持真实，除非明确的边界需要隔离
- mocks 明确且仅放置在狭窄的外部边界处
- mock 行为比其替代的真实行为更简单
- 选择的测试级别与风险匹配，且避免不必要的内部 mock
- 覆盖了成功和有意义的失败路径
- 共享状态、定时器、环境变量和全局变量在用例之间重置
- 测试文件位于代码或切片所保护的功能本地 `__tests__` 文件夹中，而非集中的包级别 `src/__tests__` 目录树
