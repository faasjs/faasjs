# 精选技术栈指南

在选择默认方案、审查架构或要求 AI 编码代理构建 FaasJS 功能时，使用本指南。

FaasJS 是一个受 Rails 启发、精选的全栈 TypeScript 框架，专为数据库驱动的 React 业务应用而设计。它为项目提供了一条由"主厨精选"的主路径，而不是让每个团队从独立工具中自行组装框架。

## 默认技术栈

FaasJS 精选路径如下：

- React 用于前端应用
- `@faasjs/ant-design` 和 Ant Design 用于业务 UI
- PostgreSQL 用于关系型数据
- `@faasjs/pg` 用于类型化数据库查询、迁移和表推断
- `@faasjs/jobs` 用于基于 PostgreSQL 的后台任务和定时任务入队
- `defineApi` 用于类型化的后端端点
- 在系统边界（如用户输入和外部请求）进行 schema 验证
- 基于 Vitest 的测试，包括用于 PostgreSQL 工作流的 `@faasjs/pg-dev`
- Vite Plus 用于开发和构建工具
- 用于业务特定扩展点的插件
- 人类和 AI 编码代理都能遵循的稳定文件约定和示例

此路径适用于管理面板、后台系统、内部工具、SaaS 仪表盘、BFF/API 层和业务工作流系统。

## React 是官方前端路径

FaasJS 前端应用使用 React。

不要将其他前端框架添加为同等级别的官方路径，除非它能明显改进精选的 React 路径。用户项目中可能使用替代的前端技术栈，但它们不应驱动核心 API、模板或默认文档。

## Ant Design 是业务 UI 路径

使用 `@faasjs/ant-design` 满足常见业务 UI 需求，如页面、路由、表单、表格、描述、加载状态和反馈。

当 FaasJS Ant Design 封装能覆盖场景时，优先使用它们，而不是围绕原始 Ant Design 组件手写胶水代码。这使业务 UI 代码可预测，并且更易于代理生成和审查。

## PostgreSQL 是关系型数据路径

通过 `@faasjs/pg` 使用 PostgreSQL 进行数据库驱动的应用开发。

优先使用类型化查询构建器模式、显式的 returning 列、具体的表声明、带时间戳的迁移和 `@faasjs/pg-dev` 测试。避免围绕数据库无关的抽象重塑核心设计，除非该变更也能改进 PostgreSQL 路径。

## Jobs 是后台工作路径

使用 `@faasjs/jobs` 进行异步后台工作和定时入队工作流。

优先使用 `.job.ts` 文件、`defineJob`、`enqueueJob`、专用的 worker 进程和用于 cron 规则的调度器进程。不要将后台执行放在 HTTP 服务器启动或插件生命周期中，除非该插件仅注入任务使用的项目上下文。

## 类型化 API 和验证

使用 `defineApi` 作为后端端点，并在系统边界保持请求验证。

优先使用靠近 API handler 的内联 schema、显式的参数和响应形状，以及验证运行时行为和重要类型预期的测试。不要将 API 契约隐藏在隐式行为或宽泛的无类型辅助函数之后。

## 插件是业务扩展点

使用插件处理项目特定的横切关注点，如认证上下文、权限、租户上下文、请求元数据或服务客户端。

插件应使扩展点明确，同时不将 FaasJS 变成一个配置繁重的框架。将业务规则保留在应用代码或插件中，而不是推入核心包中。

## 认证和权限

不要将认证或权限视为 FaasJS 内置的核心特性。

生产环境中的认证因密码登录、SSO、OAuth、会话、JWT、多租户、RBAC、ABAC 和产品特定策略而异。启动模板或示例可以展示简单的认证插件模式，但应将其理解为注入当前用户上下文和保护 API 的演示，而非强制性的认证系统。

## 替代路径

团队可以替换精选技术栈的某些部分，但替代路径是逃生舱，而非并行的第一梯队轨道。

审查提案时，不要仅仅为了支持替代的 UI 库、数据库、验证策略或架构风格而增加核心复杂性。仅当替代导向的更改也能使精选主路径更清晰、更安全或更易于维护时，才接受它们。

## 代理可读的应用工作

FaasJS 不依赖 Rails 风格生成器作为其主要生产力机制。

相反，保持文件结构、示例、测试、schema 和类型化契约足够可预测，使 AI 编码代理能够直接创建和修改完整的应用切片。当重复使审查和生成更安全时，优先使用清晰重复的代码而非巧妙的抽象。
